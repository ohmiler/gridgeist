# Direction

พื้นที่ทำงานสำหรับผู้ดูแลบัญชีที่นำเข้าบัญชีใหม่จำนวนมาก โดยจัดตามลำดับงาน `เลือกไฟล์ → ตรวจสอบ → ยืนยัน → นำเข้า → สรุปผล` ใช้น้ำเสียงแม่นยำและสงบ และให้สถานะของไฟล์กับแต่ละแถวเป็นองค์ประกอบหลักของผลิตภัณฑ์ กริดควรเป็นโครงสร้างเงียบ: หัวขั้นตอนและแถบคำสั่งอยู่ตำแหน่งคงที่ ส่วนตารางข้อมูลรับพื้นที่เด่นที่สุด

ข้อจำกัดที่ต้องรักษา: รับ CSV, validate ก่อนเขียนข้อมูล, ยกเลิกได้, retry ได้, และ partial success ต้องไม่ย้อนกลับแถวที่สำเร็จแล้วหรือส่งซ้ำโดยไม่ตั้งใจ

# Flow

1. **เลือกไฟล์** — พื้นที่อัปโหลดเป็นจุดเด่น มีปุ่ม “เลือกไฟล์ CSV”, รองรับลากวาง, บอกชนิดและขนาดที่ยอมรับโดยไม่ซ่อนใน tooltip และมีลิงก์ดาวน์โหลดตัวอย่าง CSV
2. **ตรวจสอบไฟล์** — หลังเลือกไฟล์ แสดงชื่อ ขนาด และสถานะ parsing ในบริเวณเดิมเพื่อลดการกระโดดของ layout; ผู้ใช้กด “ยกเลิก” ได้ตลอดช่วงที่ยังไม่เริ่มเขียนข้อมูล
3. **ตรวจสอบแถว** — ตารางแสดงเลขแถว ตัวระบุบัญชี ฟิลด์สำคัญ และผล validation; แถบสรุปแยก `พร้อมนำเข้า / มีข้อผิดพลาด / ซ้ำ / ข้าม` ชัดเจน ปุ่มหลักระบุจำนวนจริง เช่น “นำเข้า 184 บัญชีที่พร้อม”
4. **ยืนยัน** — ก่อนเริ่มเขียนข้อมูล สรุปผลกระทบและนโยบาย partial success: แถวที่ถูกต้องจะถูกนำเข้า แถวผิดจะไม่ถูกนำเข้า ผู้ใช้กลับไปเปลี่ยนไฟล์ได้โดยไม่สูญเสียรายงาน validation
5. **นำเข้า** — แสดงความคืบหน้าเป็นจำนวนแถว ไม่พึ่ง animation อย่างเดียว; ปุ่ม “ยกเลิก” เปลี่ยนเป็น “หยุดการนำเข้า” และอธิบายว่าแถวที่บันทึกแล้วจะคงอยู่
6. **สรุปผล** — แยก `สำเร็จ / ไม่สำเร็จ / ไม่ได้ประมวลผล` พร้อมดาวน์โหลด CSV เฉพาะแถวที่ต้องแก้ ปุ่มหลักคือ “ลองใหม่เฉพาะแถวที่ไม่สำเร็จ” และปุ่มรองคือ “นำเข้าไฟล์ใหม่”

# State contract

| สถานะ | สิ่งที่เห็น | การกระทำและการเปลี่ยนสถานะ |
|---|---|---|
| `empty` | Drop zone, ข้อกำหนดไฟล์, ตัวอย่าง CSV; ไม่มีตารางว่างหลอกตา | เลือกไฟล์ → `loading`; ปุ่มดำเนินการต่อเป็น `disabled` |
| `loading` | ชื่อไฟล์, ขั้นตอนปัจจุบัน เช่น “กำลังอ่าน 2,400 แถว”, progress ที่มีข้อความ และพื้นที่ตารางแบบ skeleton | “ยกเลิก” → `empty`; ป้องกันการเลือกไฟล์ซ้ำและ submit ซ้ำ |
| `invalid-file` | ข้อผิดพลาดระดับไฟล์เหนือพื้นที่อัปโหลด เช่น encoding, header หรือชนิดไฟล์ พร้อมเหตุผลเฉพาะ | “เลือกไฟล์อื่น” → `loading`; ไม่มีปุ่มนำเข้าและไม่สร้างรายการแถว |
| `row-error` | ไฟล์ใช้ได้ แต่ตารางทำเครื่องหมายแถวผิดด้วยไอคอน ข้อความ และฟิลด์ที่มีปัญหา; สีไม่ใช่สัญญาณเดียว | ถ้ามีแถวที่ถูกต้อง ให้ไปยืนยัน partial success ได้; ดาวน์โหลดแถวผิดหรือเลือกไฟล์แก้ไขแล้ว → `loading` |
| `disabled` | ปุ่มหลักยังมองเห็นและบอกเหตุผลใกล้ปุ่ม เช่น “ไม่มีแถวที่พร้อมนำเข้า” หรือ “กำลังตรวจสอบไฟล์” | เปิดใช้เมื่อ validation จบและมีอย่างน้อยหนึ่งแถวที่พร้อม; ห้ามอาศัย tooltip เท่านั้น |
| `success` | จำนวนสำเร็จ, เวลาเสร็จ, และทางไปดูบัญชีที่เพิ่ม; ถ้าสำเร็จทั้งหมดไม่แสดงภาษาคลุมเครือ | “ดูบัญชีที่เพิ่ม” หรือ “นำเข้าไฟล์ใหม่” → flow ใหม่ |
| `interrupted` | แบนเนอร์ระบุว่างานหยุดเพราะผู้ใช้ยกเลิก, เครือข่ายขาด หรือ session ถูกตัด พร้อมตัวเลข `สำเร็จแล้ว / ยังไม่ประมวลผล / ไม่สำเร็จ` | “ดำเนินการต่อ” ใช้ import/job ID และ retry เฉพาะรายการคงค้าง; “จบไว้เท่านี้” คงแถวสำเร็จและไปสรุปผล |

`partial success` เป็นผลลัพธ์หนึ่งของการนำเข้า ไม่ใช่ error แบบรวม: สรุปจำนวนสำเร็จและผิดแยกกัน, เก็บผลสำเร็จไว้, สร้าง retry set จาก row ID ที่ล้มเหลวหรือยังไม่ประมวลผลเท่านั้น และใช้ idempotency key เพื่อไม่สร้างบัญชีซ้ำเมื่อ retry หลัง interruption

# Responsive และ interaction

- Desktop: แถบขั้นตอนด้านบน, สรุปจำนวนกับคำสั่งด้านขวา, ตารางเป็นพื้นที่หลัก; header ตารางและแถบ action คงที่เมื่อเลื่อน
- Mobile: เปลี่ยนตารางเป็นรายการต่อแถว โดยวางสถานะและสาเหตุก่อนรายละเอียดรอง; แถบ action ติดขอบล่างแต่ไม่บังแถวหรือข้อความผิดพลาด
- ลำดับ focus เดินตามขั้นตอนงาน, error summary โฟกัสได้และลิงก์ไปยังแถวแรกที่ผิด, มี visible focus, ชื่อปุ่มชัดเจน และประกาศความคืบหน้าด้วยข้อความที่ไม่ถี่เกินไป
- การยกเลิกก่อนเขียนข้อมูลทำได้ทันที; การหยุดระหว่างเขียนต้องยืนยันผลที่จะคงอยู่และคืน focus ไปยังจุดเรียกเมื่อปิด dialog
- reduced motion ตัดการเคลื่อนไหวตกแต่ง แต่ยังคงข้อความ progress และการเปลี่ยนสถานะ

# สิ่งที่ตรวจสอบจริง

ตรวจสอบจากคำขอและเอกสาร `Gridgeist` ที่ระบุเท่านั้นว่าแบบเสนอมีเส้นทาง CSV upload, validation, cancellation, retry และ partial success และมี state contract สำหรับ `loading`, `empty`, `invalid-file`, `row-error`, `disabled`, `success`, `interrupted` ครบ รวมถึงระบุ recovery, keyboard/focus, reduced motion และ responsive transformation ตาม review checklist

ไม่มี fixture หรือหน้าจอที่รันได้ให้ตรวจ จึงไม่ได้ inspect โค้ด, render viewport, ทดลองอัปโหลดจริง, ทดสอบ keyboard/screen reader, รัน automated checks หรือยืนยัน accessibility, usability, ความปลอดภัย และความถูกต้องของ backend; รายการข้างต้นเป็นข้อกำหนดการออกแบบที่เสนอ ไม่ใช่ผล implementation หรือผล render verification
