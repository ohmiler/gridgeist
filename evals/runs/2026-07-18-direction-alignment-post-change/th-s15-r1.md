ผมจะยึดทิศทางนี้ก่อนแก้ fixture:

**Direction:** คู่มือชุมชนสำหรับคนที่กำลังเลือกกิจกรรม จัดลำดับแบบบทบรรณาธิการด้วยกริดที่มองไม่เห็น ใช้คำอธิบายเวิร์กช็อปและตัวตนผู้สอนเป็นเนื้อหาหลัก พร้อมแนวข้อมูลใช้งานจริง—วันเวลา จำนวนที่ว่าง สถานที่ การเข้าถึง และราคา—ที่สแกนได้ทันทีในจังหวะสายตารอง

สิ่งที่จะรักษาไว้ครบถ้วน:

- เวิร์กช็อปสมมติทั้งสามรายการและข้อเท็จจริงเดิม
- ป้ายกำกับว่าเป็น fictional fixture
- ตัวกรองหัวข้อและเฉพาะรายการที่ยังว่าง
- จำนวนผลลัพธ์และ empty state
- กล่องรายละเอียด รวมสถานะเต็มและปุ่มจองที่ disabled
- การจองหนึ่งที่นั่ง ข้อความยืนยัน และการคืน focus หลังปิด dialog
- การใช้งานด้วยคีย์บอร์ดและขนาดเป้าสัมผัส

ระบบภาพจะใช้พื้นผิวโทนอุ่น ตัวอักษรที่เหมาะกับการอ่านเรื่องราว ลำดับหัวเรื่องแบบสิ่งพิมพ์ และเส้นแบ่งเพียงเพื่อช่วยการอ่าน ไม่ใช้ card ซ้ำ ๆ, pills, mono labels หรือ dashboard chrome ส่วนมือถือจะเรียงใหม่เป็น “เรื่องราว → ผู้สอน → ข้อมูลตัดสินใจ → การกระทำ” แทนการย่อ desktop ลงมาตรง ๆ

จากการสำรวจจริง พบว่า fixture เป็น HTML/CSS/JS แบบ static มีสามรายการ ตัวกรองสองชนิด native `<dialog>` การจองพร้อม live status และ responsive breakpoint เดียวที่ 38rem โดยคำอธิบายเต็มของแต่ละเวิร์กช็อปปัจจุบันปรากฏเฉพาะใน dialog ยังไม่มีภาพหรือหลักฐานแบรนด์อื่นให้ยึด จึงจะไม่แต่ง testimonials งานวิจัย หรือผลลัพธ์ทางธุรกิจเพิ่มเติม

ตามข้อกำหนดของ behavioral fixture รอบนี้ ผมหยุดก่อนการแก้ไฟล์ และไม่ได้เริ่มเซิร์ฟเวอร์ เรนเดอร์หน้า หรืออ้างผลตรวจ viewport/interaction ที่ยังไม่ได้สังเกตจริงครับ
